# 推荐方案与下一步

推荐的理想写作流水线设计和3个具体行动方案。

---

## 推荐的理想写作流水线 (8阶段)

### Phase 1: 立项与构思

**目标**: 确定写什么、为谁写、怎么写

**4Agent并行头脑风暴**:
```
Agent A (市场分析): 扫榜数据 → 热门类型/题材/缺口
Agent B (创意发散): 多个核心概念 → 金手指/冲突/卖点
Agent C (结构建议): 叙事结构 → 三幕/英雄之旅/网文副本
Agent D (读者心理): 目标读者 → 爽点设计/节奏偏好

汇聚: 4个Agent的输出汇总 → 生成3-5个完整方案
人工: 从3-5方案中选择最佳 + 修改补充
```

**输出物**:
- 一句话高概念
- 类型定位 + 目标平台
- 核心金手指 + 主角设定
- 预期字数 + 更新节奏
- 核心卖点3条

---

### Phase 2: 世界观 & 角色设计

**6大类世界构建系统化清单**:
```
1. 物理/地理: 地图/气候/交通/资源
2. 社会/政治: 政权/阶级/法律/经济/势力
3. 文化/信仰: 宗教/风俗/语言/教育
4. 力量/魔法: 来源/等级/限制/代价/稀有性
5. 历史/时间: 创世/重大事件/当前纪元
6. 科技/日常: 科技水平/日常生活/通讯/医疗
```

**4层角色卡**:
```
每个主要角色(5-10人):
  Layer 1 核心: Need/Fear/Belief/Ghost
  Layer 2 外在: Want/性格标签/行为模式/语言风格
  Layer 3 关系: 对他人态度/社会角色/信息差
  Layer 4 弧线: 起点→转变事件→方向→终点
```

**力量体系设计**:
```
等级数量: 7-12大等级 × 3-9小阶段
突破条件: 资源+机缘+心境
战斗规则: 同级胜负因素
特殊能力: 每个等级解锁什么
限制: 越级挑战的合理性边界
```

**输出物**: 完整世界观文档(YAML) + 角色卡集 + 力量体系 + 势力关系图

---

### Phase 3: 大纲体系

**四级递进大纲**:

```
Level 1: 总纲 (1-2页)
  - 全书主线(三幕/多幕)
  - 分卷规划(每卷50-100章)
  - 核心转折点时间表
  - 角色弧线全局规划
  - 升级节奏全局规划

Level 2: 卷纲 (每卷1-2页)
  - 本卷目标和核心冲突
  - 本卷主要事件(10-15个)
  - 角色发展(本卷开始/结束状态)
  - 新角色引入
  - 伏笔安排(埋3-5/收1-2)
  - 字数预算

Level 3: 节纲 (每5-10章一组)
  - 小弧线结构(入场→冲突→高潮→过渡)
  - 具体事件序列
  - 爽点分布
  - 节奏设计(快慢交替)

Level 4: 章纲 (每章50-100字)
  - 核心事件(1-2个)
  - 出场角色
  - 爽点类型和位置
  - 钩子类型
  - 推进的情节线
  - 情感基调
```

**输出物**: 四级完整大纲文档 + 伏笔账本初始化 + 时间线

---

### Phase 4: 风格定义

**参考作品分析(20+维度)**:
```
输入: 5000-20000字参考文本(目标风格)
AI分析输出:
  1. 平均句长 + 标准差
  2. 段落长度分布
  3. 对话占比
  4. 描写密度
  5. 高频词统计
  6. 禁用词识别
  7. 标点习惯
  8. 比喻/修辞频率
  9. 感官分布
  10. 叙事视角和距离
  11. 时态使用
  12. 节奏模式
  13. 对话标签习惯
  14. 开头/结尾偏好
  15. 过渡方式
  16. 情感表达方式
  17. 信息密度
  18. 幽默方式
  19. 文化引用习惯
  20. 独特表达patterns
```

**输出物**:
- 风格规范文档(.md)
- 角色语音表(每人独特说话方式)
- AI味检测清单(禁用词+禁用句式)
- 参数建议(temperature/top-p)

---

### Phase 5: 章节写作

**上下文组装(每章)**:
```
自动组装:
  [Tier 1] 核心规则 + 主角状态 + 本章大纲 + 风格规则 + 禁用词 (4k)
  [Tier 2] 出场角色详细 + 伏笔列表 + 连续性注意 + 近期摘要 (4k)
  [Tier 3] RAG检索相关 + 角色语音 (3k)
  [Tier 4] 历史摘要 + 世界观节选 (2k)
  → 总计约13k tokens的丰富上下文
```

**写作闭环(3轮最大)**:
```
Round 1:
  Writer Agent → 生成初稿
  Style Reviewer → 检查风格/AI味 → 评分

Round 2 (如需):
  Continuity Reviewer → 检查一致性 → 报告问题
  Writer Agent → 修订问题段落

Round 3 (如需):
  综合审查 → 最终评分
  [评分≥80%] → 通过
  [评分<80%] → 人工介入
```

**输出物**: 定稿章节(.md)

---

### Phase 6: 生成后处理

**自动化流水线**:
```
章节定稿后自动执行:
  1. 事实提取: 从章节中提取新事实(JSON)
  2. 更新Bible: 将新事实合并到Story Bible
  3. 摘要更新: 更新L1/L2/L3三级摘要
  4. 伏笔账本: 检查本章是否有新伏笔/回收旧伏笔
  5. 角色状态: 更新出场角色的当前状态
  6. 字数验证: 确认字数在目标范围内
  7. 进度更新: 更新progress.json
```

**输出物**: 更新后的Bible + 摘要 + 状态 + 进度

---

### Phase 7: 全局审核

**定期执行(每10章/每卷结束)**:
```
审核维度:
  1. 伏笔回收: 是否有应该回收但遗忘的伏笔？
  2. 人物一致: 角色行为是否符合设定？是否有OOC？
  3. 情节弧线: 整体节奏是否合理？有无拖沓/跳跃？
  4. 去AI味: 整体AI味评分是否合格？
  5. 升级节奏: 修为/能力提升是否合理？
  6. 时间线: 有无时间矛盾？
  7. 反向修改: 如果发现问题，是否需要调整后续大纲？
```

**反向修改大纲**:
```
如果审核发现:
  - 某条情节线发展更好 → 调整大纲适配
  - 某个角色比预期更有趣 → 增加戏份
  - 某个伏笔适合提前回收 → 修改计划
  - 整体节奏偏慢 → 压缩后续章节
  
允许大纲动态调整(但保持核心主线稳定)
```

---

### Phase 8: 输出

**多平台适配**:
```
平台适配:
  起点: 每章2000-4000字 + 标准章节号 + 无敏感词
  番茄: 每章1500-3000字 + 更密集爽点 + 短句多
  知乎盐选: 单篇5000-20000字 + 完整故事 + 文学性更高
  晋江: 女频适配 + 情感线强化 + VIP章节设置
```

**敏感词过滤**:
```
自动过滤:
  - 政治敏感词
  - 色情暗示过度
  - 暴力过度
  - 平台特有禁词
  
替换策略:
  - 同义替换
  - 隐晦表达
  - 删除并补充其他内容
```

**导出格式**:
```
  - TXT(平台发布)
  - Markdown(存档)
  - EPUB(电子书)
  - PDF(打印)
```

## 行动方案

### 方案A: 基于现有项目二次开发

**核心思路**: 不重新造轮子，Fork优秀项目后组合

**技术选型**:
```
编排入口: Fork claude-novel-writeFlow
  - 4Agent并行头脑风暴
  - 3Agent写作审核闭环
  - /outline, /style, /write, /export 命令
  
审计系统: 移植InkOS的33维审计
  - 7真相文件概念
  - 多模型路由
  - 风格指纹
  
资产库: 集成Lorn.NovelWriteSkills
  - 全流程覆盖的写作规范
  - 13+平台的适配规则
  - 完整的提示模板

记忆系统: 集成Morpheus三层记忆
  - L1/L2/L3层级
  - 冲突检测
  - 轨迹回放
```

**开发步骤**:
1. Fork claude-novel-writeFlow，理解代码结构
2. 将Lorn的资产库整合为SKILL内的模板
3. 实现InkOS的审计逻辑(简化版，先做核心5维)
4. 集成Morpheus的记忆管理(可先用文件系统简化)
5. 搭建向量数据库(ChromaDB)做RAG
6. 测试完整流程

**预期时间**: 2-4周
**技术要求**: TypeScript/Python, 理解Claude Code Skill
**优势**: 起步快，有参考实现
**劣势**: 集成复杂度，各项目架构不统一

---

### 方案B: 从零构建Claude Code写作Skill

**核心思路**: 利用Claude Code原生能力，从零设计最适合的架构

**目录结构**:
```
.claude/
├── settings.json          # 权限/hooks配置
├── skills/
│   ├── 00-control/SKILL.md    # 项目总控
│   ├── 01-world/SKILL.md      # 世界构建
│   ├── 02-character/SKILL.md  # 角色设计
│   ├── 03-outline/SKILL.md    # 大纲规划
│   ├── 04-style/SKILL.md      # 风格定义
│   ├── 05-write/SKILL.md      # 章节写作
│   ├── 06-review/SKILL.md     # 审查修订
│   ├── 07-deai/SKILL.md       # 去AI味
│   └── 08-export/SKILL.md     # 导出发布
├── agents/
│   ├── brainstormer.md        # 头脑风暴Agent
│   ├── writer.md              # 写手Agent
│   ├── reviewer.md            # 审查Agent
│   └── editor.md              # 编辑Agent
└── scripts/
    ├── update-bible.js        # Bible更新
    ├── extract-facts.js       # 事实提取
    ├── check-consistency.js   # 一致性检查
    └── export-epub.js         # EPUB导出
```

**Hooks配置**:
```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write",
      "command": "node .claude/scripts/on-chapter-save.js"
    }],
    "SessionStart": [{
      "command": "node .claude/scripts/load-state.js"
    }],
    "Stop": [{
      "command": "node .claude/scripts/save-progress.js"  
    }]
  }
}
```

**文件系统状态管理**:
```
novel-project/
├── .novel/
│   ├── bible.yaml           # Story Bible
│   ├── progress.json        # 写作进度
│   ├── summaries/           # 章节摘要
│   ├── foreshadowing.yaml   # 伏笔账本
│   └── character-states/    # 角色状态
├── outlines/
│   ├── master.md            # 总纲
│   ├── vol1.md              # 卷纲
│   └── chapters/            # 章纲
├── chapters/
│   ├── ch001.md
│   └── ...
├── style/
│   ├── guide.md             # 风格指南
│   ├── voices.yaml          # 角色语音
│   └── banned-words.txt     # 禁用词
└── export/
```

**Plugin打包**:
- 最终打包为.claude-plugin格式
- 可分享给其他Claude Code用户
- 包含完整文档和使用说明

**开发步骤**:
1. 设计目录结构和数据格式
2. 编写核心SKILL.md(00-control和05-write优先)
3. 实现Agent定义(writer.md和reviewer.md)
4. 编写Hook脚本(事实提取/Bible更新)
5. 测试单章全流程
6. 逐步添加其他Skill
7. 完善和打包

**预期时间**: 3-6周
**技术要求**: Claude Code Skill格式, JavaScript, YAML
**优势**: 最契合Claude Code生态，轻量，可迭代
**劣势**: 从零开始，需要更多设计工作

### 方案C: 混合方案

**核心思路**: Claude Code Skill作为编排入口 + 外部工具做重活

**架构**:
```
┌─────────────────────────────────────────────┐
│  Claude Code Skill (编排层)                   │
│  ├── 项目管理 / 进度控制                      │
│  ├── 人工交互 / 决策点                        │
│  └── 工作流协调                               │
└──────────────┬────────────────────────────────┘
               │
    ┌──────────┼──────────────────┐
    │          │                  │
    ▼          ▼                  ▼
┌────────┐ ┌────────────┐ ┌──────────────┐
│本地Qwen│ │Claude Opus │ │ ChromaDB     │
│2.5-72B │ │关键审核    │ │ 长期记忆     │
│批量生成│ │            │ │              │
└────────┘ └────────────┘ └──────────────┘
```

**分工**:
```
Claude Code Skill: 
  - 入口命令(/write, /review, /export)
  - 上下文组装和管理
  - 人工交互和决策
  - 进度追踪

本地Qwen2.5-72B:
  - 批量章节生成(成本=电费)
  - 去AI味重写
  - 事实提取
  - 大量重复性生成任务

Claude Opus:
  - 关键章节审核
  - 复杂一致性判断
  - 创意性评估
  - 最终质量把关

ChromaDB:
  - 存储所有章节向量
  - 语义检索相关内容
  - 持久化长期记忆
  - 支持元数据过滤
```

**成本对比**:
```
方案A(纯Claude): ~$30-50/10万字
方案B(混合):    ~$5-10/10万字
方案C:          ~$3-8/10万字

成本结构:
  本地Qwen(80%生成量): 电费约$1/10万字
  Claude Opus(20%审核): ~$3-5/10万字
  ChromaDB(存储): 免费(本地)
```

**开发步骤**:
1. 部署本地Qwen2.5-72B(vLLM/Ollama)
2. 搭建ChromaDB + 设计分块策略
3. 编写Claude Code Skill(编排入口)
4. 编写Python脚本(调用本地模型+ChromaDB)
5. 实现上下文组装自动化
6. 实现生成-审核-修订循环
7. 测试完整流程

**预期时间**: 4-8周
**技术要求**: Python, Docker, 本地GPU(24GB+), Claude Code
**优势**: 成本最低，质量不低，可扩展
**劣势**: 基础设施复杂，需要GPU

## 方案选择建议

| 条件 | 推荐方案 |
|------|----------|
| 想快速开始 | 方案A (Fork+组合) |
| 追求最佳Claude Code体验 | 方案B (纯Skill) |
| 有GPU+追求性价比 | 方案C (混合) |
| 技术能力一般 | 方案B (最简单) |
| 想大批量生产 | 方案C (成本最低) |
| 想分享给他人 | 方案B (Plugin打包) |
