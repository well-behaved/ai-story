# 记忆管理与一致性技术

10种一致性维护技术的完整说明，以及推荐的混合架构方案。

---

## 1. 分层记忆架构 (MemGPT/Letta)

### 原理

模仿操作系统的内存层次结构，将LLM的有限上下文窗口类比为"RAM"，用分层管理实现理论上无限的记忆。

### 三层架构

| 层级 | 名称 | 容量 | 特点 | 类比 |
|------|------|------|------|------|
| L1 | Core Memory | 2-4K tokens | 始终在上下文中 | CPU寄存器 |
| L2 | Working Memory | 8-16K tokens | 当前任务相关 | RAM |
| L3 | Archival Memory | 无限 | 按需检索 | 硬盘 |

### 各层内容

**Core Memory (核心记忆)**:
- 角色核心设定（名字/性格/外貌/核心关系）
- 世界基础规则（力量体系/地理/时代）
- 当前主线状态（章节号/情节阶段/紧迫任务）
- 写作风格规范

**Working Memory (工作记忆)**:
- 当前场景的详细信息
- 最近2-3章的摘要
- 当前出场角色的完整状态
- 活跃伏笔列表
- 最近的情节转折

**Archival Memory (档案记忆)**:
- 全部历史章节摘要
- 完整角色档案
- 世界观百科
- 所有伏笔记录（含已回收的）
- 时间线完整记录

### 记忆管理操作

```
core_memory_append(key, value)    # 向核心记忆添加
core_memory_replace(key, old, new) # 修改核心记忆
archival_memory_insert(text)       # 存入档案
archival_memory_search(query, k)   # 搜索档案（语义）
conversation_search(query, k)      # 搜索对话历史
```

### 写作应用

- 每章开始：从Archival检索相关信息加载到Working
- 写作中：Core Memory提供基础约束
- 每章结束：从新内容提取信息存入Archival，更新Core

### 局限性

- 检索质量依赖嵌入模型
- Core Memory容量有限，信息选择是关键
- 需要精心设计何时加载/卸载信息

## 2. RAG检索增强 (Retrieval-Augmented Generation)

### 原理

将已生成内容和世界设定存入向量数据库，写作时按语义相似度检索相关片段注入上下文。

### 向量数据库选择

| 数据库 | 特点 | 适用场景 |
|--------|------|----------|
| ChromaDB | 轻量、Python原生、嵌入式 | 个人项目、原型 |
| Qdrant | 高性能、过滤强大、支持payload | 生产环境 |
| Pinecone | 全托管、无运维、按量付费 | 商业产品 |
| FAISS | Meta开源、纯向量、速度极快 | 本地、大规模 |
| pgvector | PostgreSQL扩展、SQL操作 | 已有PG基础设施 |
| LanceDB | 嵌入式、零配置、多模态 | 简单项目 |

### 分块策略

**场景分块（推荐）**:
```
Chunk 1: [Chapter 3, Scene 2] 
  "李明走进了古老的图书馆，灰尘弥漫..."
  metadata: {chapter: 3, scene: 2, characters: ["李明"], location: "图书馆", time: "第三天上午"}

Chunk 2: [Chapter 3, Scene 3]
  "夜幕降临，密室中的烛火摇曳..."
  metadata: {chapter: 3, scene: 3, characters: ["李明", "张教授"], location: "密室", time: "第三天夜"}
```

**元数据标签**:
- chapter: 章节号
- scene: 场景号
- characters: 出场角色列表
- location: 地点
- time: 故事内时间
- plot_thread: 所属情节线
- emotion: 情感基调
- importance: 重要程度(1-5)

### 检索策略

1. **语义检索**: 查询与内容的语义相似度
2. **元数据过滤**: 先按角色/地点/时间线过滤，再语义排序
3. **混合检索**: BM25关键词 + 向量语义，加权融合
4. **多维检索**: 语义 + 场景类型 + 情感匹配（AI_Gen_Novel_Style_RAG方案）

### 局限性

- 语义搜索可能遗漏"远距离"相关信息
- 分块粒度影响检索质量
- embedding模型对中文的支持参差不齐
- 无法理解因果关系和时序

## 3. 结构化Story Bible

### 原理

维护一个结构化文档（YAML/JSON），记录所有故事事实，每章生成后自动提取新信息更新。

### Story Bible结构

```yaml

world:
  name: "九州大陆"
  era: "灵气复苏后三百年"
  power_system:
    name: "灵力修炼"
    levels: ["炼气", "筑基", "金丹", "元婴", "化神", "大乘", "渡劫"]
    rules:
      - "每个大境界需要天材地宝突破"
      - "金丹期以下不可飞行"
      - "元婴期可神识外放"
  geography:
    continents: ["东洲", "西域", "南疆", "北荒", "中州"]
    current_location: "东洲·青云城"
  factions:
    - name: "青云宗"
      type: "修仙门派"
      alignment: "正道"
      notable_members: ["宗主慕容清", "大长老"]

characters:
  protagonist:
    name: "林轩"
    age: 18
    cultivation: "炼气九层"
    personality: ["坚韧", "谨慎", "重情义"]
    appearance: "黑发，剑眉星目，身材修长"
    background: "凡人出身，灵根觉醒"
    current_state:
      health: "正常"
      location: "青云宗·外门"
      equipment: ["破旧飞剑", "储物袋"]
      knowledge: ["知道父母失踪真相", "不知道宗门阴谋"]
      relationships:
        - {name: "苏灵", relation: "师姐", status: "好感"}
        - {name: "王刚", relation: "室友", status: "信任"}
    
  supporting:
    - name: "苏灵"
      role: "女主"
      cultivation: "筑基初期"
      secret: "凤凰血脉（未觉醒）"

  current_arc: "入门篇"
  chapter: 15
  active_threads:
    - {id: "T1", name: "父母失踪之谜", status: "进行中", planted: 1, expected_resolve: 50}

    - {id: "F2", description: "苏灵的异常体温", planted_chapter: 8, status: "未回收"}

  - {chapter: 1, day: 1, event: "林轩灵根觉醒"}
  - {chapter: 5, day: 30, event: "入青云宗外门"}
  - {chapter: 15, day: 180, event: "当前"}
```

### 自动更新机制

每章生成后：
1. **事实提取**: LLM从新章节提取新事实
2. **冲突检查**: 新事实与现有Bible对比
3. **更新Bible**: 无冲突则更新，有冲突则标记
4. **版本控制**: 保留Bible变更历史

### 优缺点

- **优势**: 结构清晰、可审计、可版本控制、确定性强
- **劣势**: 维护成本高、需要精心设计schema、可能遗漏隐性信息

## 4. 滚动摘要/渐进压缩

### 原理

将已生成内容逐步压缩为摘要，保持上下文窗口在限制内同时保留关键信息。

### 三级压缩

| 级别 | 名称 | 覆盖范围 | 保留度 | 容量 |
|------|------|----------|--------|------|
| L1 | 详细摘要 | 最近2-3章 | 80%细节 | 2000-4000字 |
| L2 | 中等摘要 | 最近10章 | 50%关键事件 | 1000-2000字 |
| L3 | 压缩摘要 | 全书历史 | 仅核心里程碑 | 500-1000字 |

### 摘要策略

```
写完Chapter N后:
  1. Chapter N → 详细摘要(L1)，加入L1池
  2. 如果L1池 > 5章摘要:
     - 最早的2章摘要压缩合并为L2条目
  3. 如果L2池 > 10条:
     - 最早的5条压缩合并为L3条目

上下文组装:
  [L3全书摘要] + [L2近10章摘要] + [L1最近3章详细摘要] + [当前章大纲]
```

### 关键问题

- **信息损失不可逆**: 压缩必然丢失信息
- **伏笔元素易丢失**: 早期埋的伏笔可能在压缩中消失
- **解决方案**: 伏笔/关键事实单独维护，不参与压缩

### 实际应用

- Kimi Writer: 180k触发压缩
- gemini-writer: 900k触发压缩
- RecurrentGPT: 每步更新短期记忆+长期摘要
- Long-Novel-GPT: 三阶段渐进压缩

## 5. 知识图谱

### 原理

用图数据库存储实体(节点)和关系(边)，支持复杂关系查询和约束检查。

### 图结构设计

```
节点类型:
  - Character(角色): {name, age, status, cultivation, personality}
  - Location(地点): {name, type, description, connected_to}
  - Item(物品): {name, type, power, owner}
  - Event(事件): {description, chapter, time, participants}
  - Faction(势力): {name, type, alignment, leader}

关系类型:
  - KNOWS: 角色间认识关系 + since属性
  - ALLIED_WITH / ENEMY_OF: 阵营关系
  - LOCATED_AT: 当前位置
  - OWNS: 拥有物品
  - MEMBER_OF: 从属势力
  - CAUSED: 事件因果
  - WITNESSED: 目击事件（角色视角）
```

### Neo4j查询示例

```cypher
// 查询角色当前知道的信息
MATCH (c:Character {name: "林轩"})-[:WITNESSED]->(e:Event)
RETURN e.description, e.chapter

// 查询两个角色的关系变化
MATCH (a:Character {name: "林轩"})-[r]->(b:Character {name: "苏灵"})
RETURN type(r), r.since_chapter, r.status

// 检测矛盾：角色不可能知道的信息
MATCH (c:Character {name: "林轩"})
WHERE NOT (c)-[:WITNESSED]->(:Event {id: "宗门阴谋"})
// 如果正文中林轩提到了宗门阴谋 → 矛盾！
```

### 时间属性

```
每条关系都有时间属性:
  (林轩)-[:LOCATED_AT {from_chapter: 5, to_chapter: null}]->(青云宗)
  (林轩)-[:LOCATED_AT {from_chapter: 1, to_chapter: 4}]->(凡人村)
```

### 优缺点

- **优势**: 关系查询强大、矛盾检测自然、支持时间维度
- **劣势**: 建图成本高、需要NER/RE提取、维护复杂

## 6. 上下文压缩

### 原理

通过各种技术手段，在不丢失关键信息的前提下减少上下文token数。

### 技术方法

**选择性注意**:
- 只注入与当前场景相关的信息
- 根据角色POV过滤信息（信息边界）
- 按重要程度排序，截断低重要度内容

**事实提取**:
- 从长文本中提取核心事实列表
- 去除修饰、比喻、描写，只保留"发生了什么"
- 示例：2000字场景 → 5条事实(100字)

**有损压缩**:
- 渐进式删除：先删描写→再删对话细节→再删次要事件
- 保底策略：核心事实永不压缩

**Gist Tokens (研究前沿)**:
- 将长文本压缩为少量"gist token"
- 保留语义信息但大幅减少token数
- 目前仍在研究阶段

**LLMLingua**:
- 微软的提示压缩工具
- 使用小模型判断哪些token可以删除
- 2-10倍压缩比，信息保留90%+

### 写作应用

```
上下文压缩流水线:
  原始上下文(50k tokens)
    → 选择性注意: 只保留相关角色/地点(30k)
    → 事实提取: 远期章节压缩为事实列表(15k)
    → LLMLingua: 进一步压缩(8k)
    → 最终上下文: 8k tokens包含50k信息
```

## 7. Plan-Write-Revise 分层生成

### 原理

从宏观到微观逐层规划，每层约束下一层，写完后可向上修订。

### 层次结构

```
Level 0: 核心前提 (1句话)
  "一个凡人少年在修仙世界中成长，发现自己的血脉秘密"

Level 1: 总纲 (1-2页)
  三幕结构 + 核心冲突 + 结局方向

Level 2: 卷纲 (每卷1页)
  每卷的目标 + 核心事件 + 角色发展

Level 3: 章纲 (每章1段)
  每章的起承转合 + 角色行为 + 推进的情节线

Level 4: 场景大纲 (每场景几句)
  POV + 目标 + 冲突 + 结果 + 情感变化

Level 5: 正文 (完整文本)
  基于场景大纲展开的完整叙事
```

### 约束传递

- 每层必须符合上一层的约束
- 上层修改触发下层检查/重写
- 正文不能偏离场景大纲
- 场景大纲不能违反章纲

### 修订方向

- **向下修订**: 上层改动 → 下层全部重新生成
- **向上修订**: 正文中发现更好的方向 → 反向更新大纲

## 8. 实体状态追踪

### 原理

为每个重要实体维护一个状态机，追踪其各维度的当前状态。

### 角色状态维度

```yaml
character_state:
  林轩:
    physical:
      location: "青云宗·藏经阁"
      health: 90  # 0-100
      injuries: ["左臂轻伤"]
      equipment: ["破旧飞剑", "储物袋", "玉简"]
    mental:
      emotion: "紧张"
      motivation: "寻找功法"
      stress_level: 7  # 1-10
    knowledge:  # "谁知道什么"
      knows:
        - "父母在17年前失踪"
        - "古剑中有封印"
        - "苏灵是筑基修为"
      does_not_know:
        - "宗门大长老是叛徒"
        - "苏灵有凤凰血脉"
        - "父母被困在秘境"
    social:
      reputation: "外门第一天才"
      allies: ["苏灵", "王刚"]
      enemies: ["赵天明"]
    power:
      cultivation: "炼气九层"
      techniques: ["基础剑法", "引气决"]
      special: null
```

### "谁知道什么"追踪

这是防止"全知视角污染"的关键：

```
信息边界检查:
  正文中林轩说: "大长老是叛徒！"
  检查: 林轩.knowledge.knows 中有这条信息吗？
  结果: 没有 → 矛盾！林轩不应该知道这件事
  
  除非: 本章中有某个事件让林轩获知了这个信息
```

### 状态转换规则

```
触发事件 → 状态变更:
  "林轩服用了突破丹" → cultivation: "炼气九层" → "筑基初期"
  "林轩被暗算受伤" → health: 90 → 60, injuries += "内伤"
  "苏灵告诉林轩秘密" → 林轩.knowledge.knows += "苏灵的身世"
```

## 9. 生成后一致性验证

### 原理

生成章节后，自动检查与已有事实的一致性，发现并解决矛盾。

### 验证流水线

```
生成后验证 ("小说的CI/CD"):

Step 1: 事实提取
  从新章节提取所有断言/事实
  例: "林轩使用了火焰术" → 断言: 林轩会火焰术

Step 2: 交叉引用
  将每个断言与Story Bible/知识图谱对照
  例: 检查 → 林轩.techniques 中没有"火焰术" → 潜在矛盾

Step 3: 矛盾分类
  - Hard矛盾: 直接违反已确立事实（必须修正）
  - Soft矛盾: 不太合理但可解释（标记警告）
  - 新信息: 不矛盾，是新增内容（更新Bible）

Step 4: 解决策略
  - 自动修复: 简单替换（名字/地点错误）
  - 重写段落: 涉及情节逻辑的修改
  - 人工决策: 重大方向性矛盾
  - 追溯修改: 修改前文以适配新方向
```

### "小说CI/CD"隐喻

```
类比软件开发:
  Git Commit → 章节生成
  Unit Tests → 事实一致性检查
  Integration Tests → 情节逻辑验证
  Lint → 风格一致性检查
  Code Review → 质量审查Agent
  CI Pipeline → 自动化验证流水线
  Deployment → 发布到平台
```

## 10. 多Agent架构

### 原理

用专门化的Agent分别负责不同维度的一致性维护。

### 专门化Agent分工

| Agent | 负责维度 | 维护内容 |
|-------|----------|----------|
| Writer Agent | 正文生成 | 按大纲和风格写作 |
| Continuity Agent | 前后连续性 | 时间线/因果/位置追踪 |
| Character Agent | 角色一致性 | 性格/能力/知识/关系 |
| World Agent | 世界观一致 | 规则/地理/历史/组织 |
| Plot Agent | 情节连贯 | 伏笔/冲突/弧线/节奏 |

### 协作模式

```
写作流程:
  1. Plot Agent → 确定本章情节目标
  2. Character Agent → 确认角色当前状态和可能行为
  3. World Agent → 确认场景设定和规则约束
  4. Writer Agent → 在以上约束下生成正文
  5. Continuity Agent → 验证生成内容的连续性
  6. [如有问题] → 回到Writer Agent修改
```

## 推荐混合架构

### 四层混合方案

```
Layer 1: 结构化Bible (确定性层)
  ├── YAML/JSON格式
  ├── 每章后自动更新
  ├── 版本控制(Git)
  └── 提供Core Memory内容

Layer 2: RAG检索 (语义层)
  ├── 向量数据库存储所有章节
  ├── 场景级分块 + 丰富元数据
  ├── 写作时按语义检索相关片段
  └── 提供Working Memory内容

Layer 3: 滚动摘要 (压缩层)
  ├── 三级压缩(L1/L2/L3)
  ├── 伏笔单独维护不压缩
  ├── 提供全局概览
  └── 处理超长文本

Layer 4: 生成后验证 (保障层)
  ├── 事实提取 + 交叉引用
  ├── 矛盾检测 + 自动修复
  ├── 状态更新触发
  └── CI/CD式质量门控
```

### 信息流

```
写作时上下文组装:
  [Bible核心设定 2k] + [相关角色状态 1k] + [RAG检索片段 3k] 
  + [L1近期摘要 2k] + [L2中期摘要 1k] + [L3全局摘要 0.5k]
  + [当前章大纲 0.5k] + [前一章结尾 1k]
  = 约11k tokens的丰富上下文

生成后处理:
  新章节 → 事实提取 → 更新Bible → 更新向量库 → 更新摘要 → 验证一致性
```

## Write-Agent P0-P6 分层记忆方案

Write-Agent项目提出的7级优先级记忆管理：

| 优先级 | 名称 | 内容 | 策略 |
|--------|------|------|------|
| P0 | 核心设定 | 世界观基础规则、力量体系 | 永不丢弃，始终在上下文 |
| P1 | 主线伏笔 | 核心伏笔、主要悬念 | 永不丢弃，始终在上下文 |
| P2 | 角色状态 | 主要角色当前状态 | 每章更新，始终可用 |
| P3 | 近期情节 | 最近5-10章的情节事件 | 定期压缩为摘要 |
| P4 | 场景细节 | 场景描写、环境细节 | 场景结束后压缩 |
| P5 | 对话记录 | 具体对话内容 | 保留关键对话，其余压缩 |
| P6 | 临时信息 | 生成过程中的临时数据 | 用完即弃 |

### 上下文预算分配

```
总预算: 16k tokens (假设)

P0 核心设定: 3k (固定)
P1 主线伏笔: 1k (固定)
P2 角色状态: 2k (动态，按出场角色)
P3 近期情节: 4k (滚动摘要)
P4 场景细节: 3k (当前场景)
P5 对话记录: 2k (关键对话)
P6 临时信息: 1k (工作缓存)
```

### 淘汰策略

当上下文超限时：
1. 先删P6（临时信息）
2. 压缩P5（对话→摘要）
3. 压缩P4（场景→关键点）
4. 压缩P3（近期→更简短的摘要）
5. P0和P1永不压缩

## 技术选型建议

| 场景 | 推荐技术 | 理由 |
|------|----------|------|
| 10万字以内 | Story Bible + 滚动摘要 | 简单够用 |

| 50万字以上 | 四层混合架构 | 全面保障 |

| 网文连载 | P0-P6分层 + RAG | 灵活、成本可控 |
| 最高质量 | 四层 + 多Agent验证 | 全面 |
