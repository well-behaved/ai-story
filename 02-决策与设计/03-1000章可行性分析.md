# 1000 章可行性分析

> 本文档分析支持 1000 章（200-300 万字）超长篇网文的技术可行性。
> 番茄小说热门作品动辄 1000-3000 章，超长篇支持是硬性需求。

---

## 目标

支持 **1000 章**（200-300 万字）的超长篇网文，且：

- 第 1000 章的写作质量不低于第 1 章
- 人物/世界观/伏笔的一致性不随章节增加而退化
- 每章写作时的上下文加载时间恒定
- 状态文件大小可控，不无限膨胀

### 关键观察

- 能支持 1000+ 章的项目（InkOS、天命）都有 **快照式设计**（状态是当前值，不是历史记录）
- 不能支持的项目都是 **追加式设计**（不断在文件末尾添加内容）
- 分界线清晰：**快照 vs 日志** 决定了上限

## 5 个核心难题和解法

### 难题 1：摘要膨胀

**问题**：每章写完都要记录摘要，1000 章 = 1000 条摘要，即使每条 100 字也有 10 万字

**解法：分层压缩**

```
分层策略：
├── 最近 10 章：详细摘要（200-300字/章）→ 约 2500 字
├── 11-50 章：中等摘要（50-100字/章）→ 约 3000 字
├── 51-当前卷剩余：一句话摘要（20字/章）→ 约 1000 字
├── 往期卷：卷级摘要（500字/卷）→ 约 5000 字（10卷）
└── 总计注入量：约 11500 字 = 恒定
```

**压缩时机**：
- 每写完 1 章：更新最近 10 章摘要，第 11 章自动降级为中等摘要
- 每写完 50 章：51+ 章降级为一句话摘要
- 每完成 1 卷：上一卷压缩为卷级摘要

### 难题 2：伏笔断线

**问题**：第 50 章埋的伏笔，到第 500 章还没回收，AI 早就忘了

**解法：紧迫度分级 + 定期盘点**

```
伏笔生命周期：
├── 潜伏（刚埋下，暂不处理）
├── 铺垫（开始出现呼应线索）
├── 可回收（条件成熟，可以回收）
├── 紧急回收（超过预期回收章节，必须尽快处理）
└── 已回收（完成，归档）
```

**机制**：
- 每章写作时注入"紧迫度 >= 铺垫"的伏笔列表
- 每 50 章做一次伏笔盘点：哪些该升级紧迫度？哪些该回收？
- 伏笔账本只记录"活跃伏笔"，已回收的移入归档区不再加载

**注入量控制**：
- 活跃伏笔通常 10-30 条
- 每条 30-50 字描述
- 总计 500-1500 字，可控

### 难题 3：角色档案膨胀

**问题**：1000 章可能出场 200+ 个角色，全部加载不现实

**解法：按需加载（只注入本章出场角色）**

```
角色分层：
├── 主角团（3-5人）：永远加载核心状态
├── 重要配角（10-20人）：出场时加载
├── 普通角色（50-100人）：出场时加载精简版
└── 龙套（100+人）：仅保留名字和一句话描述
```

**机制**：
- 章纲中标注本章出场角色列表
- Planner 根据列表从角色目录中提取相关档案
- 角色状态文件是"当前快照"：只记录最新状态，不记录历史
- 历史状态在章节摘要中间接保留

**注入量控制**：
- 主角 × 3 × 200字 = 600字
- 配角 × 2 × 100字 = 200字
- 总计约 800-1500 字

### 难题 4：大纲文件过大

**问题**：1000 章的章纲如果放一个文件，预估 20-30 万字，根本无法加载

**解法：分卷管理**

```
大纲目录结构：
02-大纲/
├── 总纲.md（全书级别，2000-3000字）
├── 卷01/
│   ├── 卷纲.md（本卷概述，1000-2000字）
│   └── 章纲/
│       ├── 第001章.md（200-300字）
│       ├── 第002章.md
│       └── ...第100章.md
├── 卷02/
│   ├── 卷纲.md
│   └── 章纲/
└── ...卷10/
```

**每章写作时加载**：
- 总纲（2000字）+ 本卷卷纲（1500字）+ 本章章纲（300字）= 约 3800 字
- 不加载其他卷的任何内容
- 1000 章和 10 章，这部分加载量完全一样

### 难题 5：累积误差

**问题**：AI 每章产生微小的一致性偏差，1000 章后累积成灾难性错误

**解法：定期全局审查**

```
审查频率：
├── 每章：Reviewer 检查与前文一致性（自动）
├── 每 10 章：Tracker 更新角色状态/世界状态（自动）
├── 每 50 章：全局伏笔盘点 + 时间线校验（半自动）
├── 每 100 章（卷末）：全书一致性扫描（需用户参与）
└── 异常触发：当 Reviewer 评分连续 3 章低于 35 分时强制全局审查
```

**审查内容**：
- 角色性格是否偏移（对比初始设定）
- 力量体系是否自洽（等级/能力/限制）
- 时间线是否矛盾（日期/年龄/事件顺序）
- 地理信息是否一致（地名/距离/方位）
- 伏笔是否遗漏（超期未回收的标记为紧急）

## 架构设计原则（确保 1000 章可行）

### 原则 1：状态文件是"当前快照"而非"追加日志"

```
❌ 追加日志式（会膨胀）：
角色状态.md:
  第1章：张三实力为炼气一层
  第5章：张三突破到炼气二层
  第20章：张三突破到筑基
  ...第999章：张三突破到大乘
  → 1000行，持续增长

✅ 快照式（恒定大小）：
角色状态.md:
  张三：
    当前境界：大乘期
    最近变化：第995章突破（从合体期升级）
    核心关系：[列表]
    当前目标：[描述]
  → 永远只有最新状态，约200字/角色
```

### 原则 2：每章注入的上下文大小恒定

目标：约 5-6K token（10000-12000 字），无论是第 1 章还是第 1000 章。

```
上下文预算：
├── P0 骨架（3000字）：设定摘要 + 总纲 + 卷纲 + 章纲
├── P1 角色（1500字）：本章出场角色当前状态
├── P2 衔接（2500字）：前1章完整正文
├── P3 伏笔（1000字）：活跃伏笔列表
├── P4 上下文（1500字）：最近5章摘要
├── P5 约束（500字）：风格规范 + 禁用词
└── 总计：约 10000-12000 字 ≈ 5-6K token
```

### 原则 3：所有文件支持分卷

- 大纲：每卷一个目录
- 正文：每卷一个目录
- 章节摘要：每卷一个文件（旧卷只保留卷级摘要）
- 审查报告：每卷一个目录

单文件不超过 **5000 字**（确保 Claude Code 可以一次性 Read）。

### 原则 4：增量更新，不全量重写

```
每章写完后的更新操作：
├── 角色状态.md → Edit（修改变化的角色字段，不重写整文件）
├── 伏笔账本.md → Edit（更新状态变化的伏笔，不重写）
├── 章节摘要.md → Append（添加本章摘要到末尾）
├── 世界状态.md → Edit（只在世界发生变化时更新）
└── 情节线.md → Edit（更新当前情节线进度）
```

### 原则 5：定期压缩旧数据

```
压缩策略：
├── 章节摘要：每50章压缩一次旧摘要
├── 伏笔账本：已回收伏笔移入归档文件
├── 角色状态：退场角色移入归档文件
├── 审查报告：旧卷报告压缩为摘要
└── 正文：正文不压缩（用户需要全文）
```

## 1000 章场景模拟

### 写第 1 章时

```
加载：设定(1000) + 总纲(2000) + 卷01纲(1500) + 章001纲(300) + 主角档案(600)
     + 风格规范(500)
总计：约 5900 字 ≈ 3K token
状态文件：全部为初始状态，很小
```

### 写第 100 章时

```
加载：设定(1000) + 总纲(2000) + 卷01纲(1500) + 章100纲(300)
     + 出场角色(1200) + 前1章正文(2500) + 活跃伏笔(800)
     + 最近5章摘要(1500) + 风格规范(500)
总计：约 11300 字 ≈ 5.5K token
状态文件：角色状态约3000字，伏笔约2000字
```

### 写第 1000 章时

```
加载：设定(1000) + 总纲(2000) + 卷10纲(1500) + 章1000纲(300)
     + 出场角色(1200) + 前1章正文(2500) + 活跃伏笔(1000)
     + 最近5章摘要(1500) + 风格规范(500)
总计：约 11500 字 ≈ 5.7K token
状态文件：角色状态约5000字（活跃角色），伏笔约3000字（活跃伏笔）
```

**结论**：第 1000 章的上下文加载量与第 100 章基本一致，设计目标达成。

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 摘要压缩丢失关键信息 | 中 | 高 | 压缩时保留关键事件标记，审查时可回溯原文 |
| 伏笔分级判断失误 | 中 | 中 | 用户可手动调整紧迫度，定期盘点兜底 |
| 角色按需加载遗漏 | 低 | 中 | 章纲中强制标注出场角色，Reviewer检查一致性 |
| 分卷文件管理混乱 | 低 | 低 | 严格的命名约定 + Planner自动维护 |
| 累积误差在审查间隙爆发 | 中 | 高 | 缩短审查间隔（50章→30章），低分预警 |

---

## 结论

**1000 章完全可行**，前提是遵循以上 5 条架构原则。核心思想是：

> 用 **快照 + 分层 + 分卷 + 按需加载 + 定期压缩** 确保系统复杂度不随章节数增长。

这不是理论推导——InkOS（3000章）和天命（3000章）已经用不同的技术方案验证了可行性。我们的方案借鉴了它们的设计思想，并适配到 Claude Code Skill 的技术栈上。
