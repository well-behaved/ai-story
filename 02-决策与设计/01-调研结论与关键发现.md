## 十大关键发现

### 1. 上下文保持是第一难题

不取决于模型能力，取决于记忆系统设计。所有成功的长篇方案都有外部记忆层：

- **Story Bible**（故事圣经）：角色、世界观、规则的权威来源
- **真相文件**（Truth Files）：InkOS 的 7 个核心状态文件
- **分层摘要**：近章详细 → 远章压缩 → 全局一句话

没有外部记忆的方案，普遍在 20-30 章后出现严重的一致性崩溃。上下文窗口再大也没用——问题不是"装不装得下"，而是"该装什么"。

### 2. 多 Agent 流水线是行业共识

所有成熟项目都采用专门化 Agent 分工：

| 项目 | Agent 数量 | 核心模式 |
|------|-----------|---------|
| InkOS | 10 | 导演→编剧→作家→编辑→审计... |
| BMAD | 8 | 超级架构师→多角色切换 |
| writeAgent | 4 | planner→writer→reviewer→polisher |
| claude-novel-writeFlow | 3 | 创意→写作→编辑三智能体循环 |
| oh-story-claudecode | 多级 | 子Skill分工+Hooks协调 |

核心模式是 **Planner → Writer → Reviewer → Reviser** 闭环。单 Agent 方案（只有一个"万能作家"）在质量上远逊于分工方案。

### 3. 审核/修订闭环是质量保证的核心

没有反馈循环的系统质量快速退化。关键实践：

- **InkOS**：33 维审计矩阵，每章必过
- **writeAgent**：10 维 50 分制评分，低于 35 分自动重写
- **claude-novel-writeFlow**：三智能体 3 轮迭代，每轮聚焦不同维度
- **oh-story-claudecode**：六大门禁逐级过滤

规律：**没有审核的 AI 写作 = 垃圾生成器**。审核环节的投入应该与写作环节等量甚至更多。

### 4. Claude Code Skill 生态有空白机会

- **oh-story-claudecode**（997 stars）刚发布就爆了——说明需求巨大但供给不足
- 目前 Claude Code Skill 市场上的写作工具极少
- 大多数 AI 写作项目是独立 Web 应用或 Python 脚本，没有利用 Claude Code 的原生能力
- Claude Code 的子 Agent + Hooks + 文件操作天然适合写作工作流
- 先发优势窗口约 3-6 个月

### 5. 番茄小说是最适合 AI 的平台

- **公式化结构**：系统文、升级文、重生文都有成熟套路
- **快节奏**：每章 2000-4000 字，必须有钩子
- **强钩子**：章末悬念是硬性要求，AI 按模板生成效果好
- **日更压力**：作者对效率有刚需
- **读者对文学性要求不高**：追求爽感和节奏，不追求文笔
- **免费+广告模式**：量大于精，适合 AI 辅助批量生成

对比起点中文网（质量要求高、读者挑剔）和知乎盐选（短篇为主），番茄是 AI 写作最佳落地场景。

### 6. 去 AI 味必须分层处理

只换禁用词是表面功夫。oh-story-claudecode 的**六大门禁**是目前最成熟方案：

1. **禁用词过滤**：删除"值得注意的是""不禁""竟然"等高频AI词
2. **句式打散**：消除"主语+状语+谓语"的单调结构
3. **心理外化**：把"他感到不安"改成具体动作/环境映射
4. **节奏打碎**：交替使用超短句和长句，避免均匀的中等长度
5. **对话去腔调**：消除"嗯，你说的对"式的礼貌客套
6. **结尾去升华**：删掉章末的哲理总结和感悟段落

**每一层都独立检查**，不是一次性处理。这是 AI 味检测的递进漏斗。

### 7. 40% 规划 + 20% 生成 + 40% 编辑是最佳比例

来自多个项目的实验结论：

- 花 80% 时间在生成上的结果最差（无规划 = 失控）
- 纯规划不生成也没用（完美的大纲不等于好的小说）
- **最佳实践**：
  - 40% 时间做规划（大纲、角色、设定、章纲）
  - 20% 时间做生成（在精确章纲指导下写作）
  - 40% 时间做编辑（审查、修订、去AI味、润色）

这意味着我们的 Skill 应该在规划和编辑环节投入更多设计。

### 8. 分层文件记忆足以支撑超长篇

不需要向量数据库 / Neo4j / 任何额外基础设施：

- **InkOS**（6k stars）用 7 个 Markdown/JSON 文件 + SQLite 管理 3000 章
- **天命 tianming** 用 12 维快照实现 O(1) 复杂度查询
- **writeAgent** 用 P0-P6 优先级截断控制上下文大小

关键设计：
- 文件系统即数据库
- 状态文件是"当前快照"而非"追加日志"
- 上下文大小恒定，不随章节数增长
- 按需加载，不全量注入

### 9. 网文方法论有独立的知识体系

不是西方叙事理论（三幕式、英雄之旅）的翻版，需要专门处理：

- **黄金三章**：前三章决定生死，必须完成人设+冲突+金手指展示
- **爽点设计**：打脸/升级/获宝/复仇，有固定节奏公式
- **升级体系**：力量等级、境界划分、每N章一次突破
- **副本结构**：任务→挑战→收获→下一个任务的循环
- **伏笔账本**：埋伏笔和回收伏笔的系统管理
- **章末钩子**：悬念/反转/新信息，每章必须有
- **注水检测**：读者对灌水极其敏感，AI 容易犯这个错

这些方法论需要作为参考资料库内置到 Skill 中。

### 10. Qwen2.5-72B 是中文网文最佳开源模型

- 中文表达自然度最高
- 对网文套路的理解最到位
- 72B 版本在创意写作上接近 GPT-4

**但对于我们的方案**，用 Claude 做全链路更简单：
- Claude Code 原生支持，不需要搭建推理服务
- Claude 的指令遵循能力更强（对 Skill/Agent 指令的执行更精确）
- 成本换效率，适合工具开发者（而非大规模内容农场）

## 调研结论一句话总结

> **用 Claude Code Skill 做一个面向番茄小说的写作工具，采用 4-5 个 Agent 流水线 + 分层文件记忆 + 审核闭环 + 去AI味门禁，是当前生态中 ROI 最高的方向。**
