# 技术选型决策

> 本文档记录所有关键技术决策，包括选择理由和否决理由。
> 每个决策都基于 [调研结论](01-调研结论与关键发现.md) 中的数据支撑。

---

## 决策 1：目标平台 → 番茄小说

### 选择理由

1. **免费/广告模式**：不需要读者付费，按阅读量分成，量大于精
2. **日更 4000-6000 字**：每天稳定产出，AI 辅助效率提升最明显
3. **快节奏强钩子**：结构化程度高，AI 按模板生成效果好
4. **公式化结构**：系统文/升级文/重生文都有成熟套路可循
5. **读者对文学性要求相对宽容**：追求爽感和节奏，不苛求文笔
6. **市场巨大**：番茄小说日活 8000 万+，对内容供给有持续需求
7. **AI 友好**：平台对 AI 辅助创作态度相对开放（需保持关注政策变化）

### 否决选项

| 平台 | 否决理由 |
|------|---------|
| 起点中文网 | 质量要求太高，读者极其挑剔，精品路线不适合 AI 辅助批量产出 |
| 知乎盐选 | 以短篇为主（1-5万字），不需要长篇记忆方案，且风格文学性要求高 |
| 晋江文学城 | 女频为主，情感细腻度要求高，AI 目前在这方面较弱 |
| 刺猬猫 | 受众小，二次元向需要极深的亚文化理解 |
| 海外平台（Webnovel） | 需要英文写作，市场不熟悉，不是第一优先级 |

### 决策依据

- 调研发现 #5：番茄小说是最适合 AI 的平台
- Lorn.NovelWriteSkills 已有番茄平台适配模板
- oh-story-claudecode 明确以番茄为目标平台

## 决策 2：技术路线 → Claude Code Skill

### 选择理由

1. **原生支持子 Agent**：Claude Code 的 Agent 机制天然适合多Agent流水线
2. **原生 Hooks 支持**：session-start/pre-compact/post-compact 精确满足写作工作流需求
3. **原生文件操作**：Read/Write/Edit/Glob 直接操作项目文件，不需要额外适配
4. **不需要额外基础设施**：不需要数据库、不需要服务器、不需要 Docker
5. **可快速迭代**：改 Markdown 文件即部署，无需编译/打包
6. **已有成功先例**：oh-story-claudecode（997 stars）证明了 Claude Code Skill 做写作工具的可行性
7. **用户已有 Claude Code**：目标用户本身就是 Claude Code 用户，零安装成本

### 否决选项

| 方案 | 否决理由 |
|------|---------|
| 独立 Web 应用 | 工程量大 10 倍，需要前端+后端+数据库+部署，维护成本高 |
| Dify/Coze 工作流 | 灵活度不够，节点间数据传递受限，无法精细控制文件操作 |
| LangChain/CrewAI 框架 | 需要 Python 环境，增加了依赖和复杂度，不如 Claude Code 原生方案简洁 |
| 本地开源模型 | 质量不如 Claude（即使 Qwen2.5-72B），且需要 GPU 服务器 |
| VS Code 插件 | 开发成本高，需要学习 VS Code API，用户群更窄 |
| ComfyUI 式节点编辑 | 过于可视化，对写作流程来说是过度设计 |

### 决策依据

- 调研发现 #4：Claude Code Skill 生态有空白机会
- 调研发现 #2：多 Agent 流水线是行业共识（Claude Code 原生支持）
- oh-story-claudecode 验证了完整的 Skill 架构可行性

## 决策 3：记忆方案 → 分层文件记忆

### 选择理由

1. **文件系统即数据库**：Claude Code 原生 Read/Write/Edit 直接操作，无需中间层
2. **可人工审查和修改**：所有状态文件都是 Markdown，用户随时可以看到和修改，不是黑箱
3. **InkOS 已验证**：用 7 个文件管理 3000 章的超长篇，证明文件方案的上限足够高
4. **零安装成本**：不需要安装任何数据库、不需要 Docker、不需要配置连接
5. **上下文大小恒定**：300 章和 3 章注入的上下文大小一样（因为是当前快照 + 增量更新 + 压缩）
6. **Git 友好**：所有状态变化可追踪，可回滚，可对比
7. **跨平台**：文件系统在 Windows/Mac/Linux 上行为一致

### 分层设计

```
记忆层次：
├── L0 永久层：设定/世界观/角色基础档案（几乎不变）
├── L1 卷级层：卷纲/卷内角色弧（每卷更新一次）
├── L2 动态层：角色状态/伏笔账本/世界状态（每章更新）
├── L3 缓冲层：最近5章摘要（滑动窗口）
└── L4 临时层：当前章写作上下文（用完即弃）
```

### 否决选项

| 方案 | 否决理由 |
|------|---------|
| 向量数据库 RAG（Pinecone/Milvus） | 过重；番茄类型文结构固定，不需要语义检索；增加部署复杂度 |
| 知识图谱 Neo4j | 过度工程化；关系查询可以用文件+命名约定解决；安装配置成本高 |
| 纯上下文窗口 | 30 章后必崩；即使 200K 窗口，装不下 1000 章的信息且效率低 |
| SQLite 数据库 | 比纯文件好一些，但增加了复杂度，Claude Code 操作 SQL 不如直接操作文件自然 |
| Redis 缓存 | 需要额外服务，且写作场景不需要高并发读写 |

### 决策依据

- 调研发现 #8：分层文件记忆足以支撑超长篇
- 调研发现 #1：上下文保持的关键是记忆系统设计，不是模型能力
- InkOS + writeAgent 的实践验证

## 决策 4：Agent 数量 → 4-5 个

### Agent 角色定义

| Agent | 职责 | 对标来源 |
|-------|------|---------|
| Planner | 规划章纲、管理大纲、维护记忆文件 | writeAgent.planner |
| Writer | 根据章纲和上下文写正文 | writeAgent.writer + writeFlow.creator |
| Reviewer | 10维审查评分、判断是否需要重写 | writeAgent.reviewer + writeFlow.editor |
| Polisher | 润色文笔、去AI味、番茄平台适配 | oh-story.deslop |
| Tracker | 伏笔/时间线/角色状态追踪、一致性检查 | InkOS.continuity-auditor |

### 选择理由

1. **writeAgent 的 4 个 Agent** 职责分明，已验证有效
2. 加一个 **Tracker**（一致性检查器）处理伏笔/时间线，这是长篇刚需
3. 控制在 5 个以内，每个 Agent 的职责清晰，不会出现职责模糊
4. Claude Code 的子 Agent 调用有开销，过多 Agent 会显著增加延迟和成本

### 否决选项

| 方案 | 否决理由 |
|------|---------|
| 10 个 Agent（InkOS 级别） | 对 Claude Code 来说太重，Agent 间协调复杂度呈指数增长 |
| 12 个 Agent（creative-writing-skills） | 需要额外框架支持，Claude Code 原生不适合管理这么多Agent |
| 单 Agent | 没有审核闭环，质量无法保证，违反调研发现 #3 |
| 2 个 Agent（Writer + Reviewer） | 缺少规划和记忆管理，长篇会失控 |

### 决策依据

- 调研发现 #2：多 Agent 流水线是行业共识
- 调研发现 #3：审核/修订闭环是质量保证的核心
- 控制复杂度的工程判断

## 决策 5：借鉴组合策略 → 不 Fork 任何一个，组合借鉴

### 策略说明

不直接 Fork 任何现有项目，而是从多个项目中提取最佳实践组合成新方案。理由：

1. 没有任何单一项目完全满足我们的需求
2. 每个项目都有明显短板
3. 组合借鉴可以取各家之长，避各家之短
4. 从零构建确保架构统一性，避免 Fork 后的技术债

### 借鉴矩阵

| 模块 | 借鉴来源 | 借鉴内容 | 改进方向 |
|------|---------|---------|---------|
| **骨架** | oh-story-claudecode | Skill/Agent/Hooks 组织方式、目录结构 | 简化层级，增加分卷支持 |
| **记忆** | InkOS 真相文件 + writeAgent P0-P6 | 7核心文件 + 优先级截断 | 改为5核心文件，去掉JSON用纯Markdown |
| **写作** | claude-novel-writeFlow + writeAgent | 三智能体循环 + 10维审查 | 合并为 Writer→Reviewer→Polisher 链 |
| **去AI味** | oh-story-claudecode | 六大门禁分级处理 | 增加番茄平台特化规则 |
| **番茄特化** | Lorn.NovelWriteSkills | 13平台适配模板、题材模板 | 聚焦番茄一个平台，做深做透 |
| **超长篇** | InkOS + 天命 tianming | 分层压缩 + 快照设计 | 适配 Claude Code 文件操作方式 |

### 各项目的短板（为什么不直接 Fork）

| 项目 | 短板 |
|------|------|
| oh-story-claudecode | 追踪文件无压缩机制，1000章会膨胀；文档不够详细 |
| writeAgent | 单文件 outline 架构，100章以上就需要重构 |
| claude-novel-writeFlow | 没有番茄平台特化，没有去AI味方案 |
| InkOS | 不是 Claude Code Skill，需要完全重写架构 |
| Lorn.NovelWriteSkills | 模板为主，缺少写作引擎和记忆方案 |

## 决策总结

```
目标平台：番茄小说
技术路线：Claude Code Skill
记忆方案：分层文件记忆（5核心文件）
Agent数量：5个（Planner/Writer/Reviewer/Polisher/Tracker）
借鉴策略：组合借鉴，不Fork
```

所有决策均有调研数据支撑，且经过否决选项对比论证。
